# 0001. Architecture Style: Modular Monolith

## 1. Контекст
Проєкт знаходиться на етапі MVP (Minimum Viable Product). Команда невелика (1-3 розробники). Основними доменами є "Каталог", "Запаси" та "Управління Користувачами". Головна вимога – швидкість розробки та простота розгортання.

## 2. Рішення
На поточному етапі обираємо **Модульний Моноліт**.
Усі Bounded Contexts (Каталог, Запаси, Користувачі) будуть реалізовані в межах однієї кодової бази, але з суворим розділенням на модулі/папки (api/, service/, domain/ для кожного контексту).

## 3. Альтернативи
* **Пілот мікросервісів:** Відхилено, оскільки додає складність розгортання, міжсервісних викликів та моніторингу, що є надлишковим для MVP.

## 4. Наслідки
* **Позитивні:** Спрощене розгортання (один артефакт), легке рефакторинг, швидка розробка.
* **Негативні:** З часом, зі зростанням команди та кодової бази, моноліт може стати некерованим.

## 5. Умови перегляду
Рішення про перехід на мікросервіси буде переглянуто, коли:
* Команда розробки зросте до 5+ осіб.
* Час збірки (build time) перевищить 10 хвилин.
* Окремі домени (наприклад, "Запаси") матимуть радикально різні вимоги до масштабування (наприклад, потреба в 5-кратно більшій кількості інстансів, ніж "Каталог").
